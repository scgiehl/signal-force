# Voice and Style

The voice rule every Signal Force document follows. Pulled forward from `signal_force_foundation.md` and the existing golden files. Read this before writing.

---

## The one-line rule

> Write like an experienced GovCon operator briefing a BD director who has to make a decision Monday morning.

If the file doesn't read like that, rewrite it.

---

## Voice

- **Direct.** Lead with the point. Bury caveats only if they are actually material.
- **Operational.** "What do I do with this on Monday?" should be answerable from every section.
- **GovCon-specific.** Use the actual language federal contracting uses — agencies, vehicles, NAICS, PSCs, set-asides, color reviews, B&P, capture, recompete, FCL, CPARS, PIRs. Don't define common terms in-line; use the glossary.
- **Analytical, not promotional.** State what is observable, what is inferred, and the basis for the inference. Don't market.
- **Quiet authority.** No exclamation, no superlatives, no consultant tics. The reader is smart; respect them.

---

## What to avoid

- Generic business writing ("In today's rapidly evolving landscape...")
- Buzzwords without operational meaning ("synergy", "next-gen", "best-in-class", "transformative")
- Hedge stacking ("could potentially possibly indicate that...")
- High school definitions of things every GovCon BD person already knows
- Marketing voice imported from capability decks
- Cheerful conclusions that don't follow from the analysis
- "We" / "you" / "our" unless the document genuinely speaks for Signal Force as a service

---

## Structure rules

Every intelligence library file should include, in order:

1. **Title and library/module identifier** at line 1.
2. **Version + date + library + module + intended use** block immediately after.
3. **File purpose** — one paragraph stating what this file teaches the AI to do.
4. **Why it matters** — the BD consequence of getting this right or wrong.
5. **Core concepts** — the substantive content. Subsections as needed.
6. **Key terms** — terms specific to this file; link to glossary for shared terms.
7. **Practical examples** — at least three. Real, specific, GovCon.
8. **Research methods + data sources** — how the user (or Signal Force) keeps this file current.
9. **How to apply** — what BD action this enables.
10. **BD and capture implications** — strategic conclusions.

If a section is genuinely not relevant, drop it explicitly and say why; don't silently omit.

---

## Length and depth

Default file length: long. The golden file for market intelligence states "avoid shallow summaries." Most existing intelligence files are 30–80 KB. Match that depth. A 5 KB file is almost always evidence the topic wasn't covered, not evidence of efficiency.

Exception: foundational/spec files (this one, file_naming_standard, glossary) should be tight.

---

## Formatting conventions

- Use `#`, `##`, `###` headings. Don't skip levels.
- Use tables for comparisons, lists of structured fields, and matrices.
- Use code fences for templates, scripts, and ASCII diagrams.
- Use `>` blockquotes sparingly — reserve for quoted source language or boxed callouts.
- Prefer bulleted lists with bold lead-ins for scannability.
- No emoji.
- No "Signal Force Federal Market Intelligence Library / [Module]" as the H1 — use the module name as H1 and put the library identifier in the metadata block below.
